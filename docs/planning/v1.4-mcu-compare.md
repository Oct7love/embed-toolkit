# v1.4 — MCU 选型对比器（mcu-compare）planning

路由：`/tools/hardware/mcu-compare`

> 本文遵循 karpathy guidelines 阶段 1：先表态再动手。等用户确认后才开工。

---

## 1. 已知事实（从代码读到的，不是猜的）

- `public/chips/index.json` 当前 schema **只有** 6 个字段：`id / name / series / manufacturer / package / pinCount`。**没有** cpu / 主频 / Flash / RAM / 电压 / 价格等信息。
- 各系列 JSON（`stm32f4.json` 等）只存 pin 级数据（`pins[]`），**没有** chip 级 spec。
- 共 45 款芯片，按 series 分到 7 个文件 + index。
- `scripts/generate-chips.ts` 是一次性生成 pin 数据的脚本，可作为扩字段的脚本模式参考。

## 2. 假设

- A1：补 chip-level spec 字段写进 **`index.json`**（不写进各 series 文件，避免让 GPIO planner 也吃 spec 数据增加 fetch payload）。
- A2：用户基本只对比 2-4 款芯片，UI 不做 5+ 款（雷达图叠 5 条以上不可读）。
- A3：芯片"差异表格"显示 **不同的字段**（数值不同则高亮），相同字段折叠或淡化。
- A4：价格在国内电商波动剧烈、不同封装价差大、且属于厂商敏感信息——**不收录单一价格数字**。
- A5：URL state 用 `?ids=stm32f103c8t6,stm32f407vgt6` 多选，不超过 4 个。

## 3. 开放问题（需你拍板）

### Q1 — 雷达图的"性能"维度怎么定义？

| 选项 | 计算 | 优点 | 缺点 |
|------|------|------|------|
| **(a) 主频归一化** | `maxFreq / 480` × 100 | 简单透明 | 对 Cortex-M0 不公平（M4 100MHz ≠ M0 100MHz）|
| **(b) 主频 × 架构系数** | M0=1.0 / M3=1.25 / M4=1.5 / M7=2.14（DMIPS/MHz 比） | 接近 DMIPS 真实值 | 系数源自 ARM 官方文档但仍是粗估 |
| **(c) 直接收录 DMIPS 数值** | datasheet 抄录或填 null | 最准 | 数据采集成本高，且很多 datasheet 没标 |

→ **倾向 (b)**：透明、可解释、不编造，雷达图 "Performance" 轴标 `MHz × DMIPS/MHz`。请确认。

### Q2 — 雷达图的轴选什么？

候选维度：Performance（见 Q1）/ Flash / RAM / GPIO 数 / 外设丰富度 / 低功耗（Sleep current）/ 价位段。

→ **倾向 4 轴**：`Performance / Flash / RAM / Peripherals`。低功耗和价位都不可靠（前者数据少，后者敏感）。请确认是否需要 5/6 轴。

### Q3 — 字段不确定时怎么处理？

→ **方案**：JSON 字段缺失或 `null` → UI 渲染为 `"—"` 并加 tooltip "未收录"。绝不为了让雷达图好看而编造数字。请确认这是 hard rule。

### Q4 — "Peripherals" 字段的具体类型？

候选：(a) 整数 `peripheralCount: 23` 一个总数；(b) 对象 `{ uart: 3, spi: 2, i2c: 2, can: 1, usb: true, eth: false, fs_mc: false }`。

→ **倾向 (b)**：差异表格能逐项展示（"F407 比 F103 多了 OTG_HS"），雷达图的 Peripherals 轴用对象的字段数加权和。请确认。

### Q5 — features 标签做什么用？

候选：(a) 卖点标签 `["FPU", "DSP", "USB-OTG-HS", "Ethernet"]`，差异表 + 雷达图旁边显示；(b) 不做。

→ **倾向 (a)**：标签是用户选型最直观的"卖点"，比一堆数字有用。请确认。

## 4. 简化方案考察

### S1：能不能不用 Recharts 雷达图？

→ Recharts 已集成（其他工具用了 Composed/Line），雷达图同包不增加 bundle。值得做。

### S2：能不能不补字段，只显示已有的 pinCount/package/series？

→ 不行。pinCount/package 不足以做"选型对比"——用户要看 Flash/RAM/主频。补字段是工具的核心价值。

### S3：能不能不做 URL state，只用 store？

→ 能。但 URL state 是这工具的天然分享机制（"老王你看我对比了这俩，链接发你"）。**保留 URL state**，但实现得最小：`useSearchParams + replace`，不入 store。

## 5. 数据扩字段执行方案

写一个独立小脚本 `scripts/enrich-chip-specs.ts`，按 series 维度查官方 datasheet 一次性 hardcode 进去。**不动 pin 数据，只动 index.json 的 chip 对象**。每条芯片新增字段：

```ts
{
  // 现有字段保持不变 ↓
  id, name, series, manufacturer, package, pinCount,
  // 新增字段 ↓
  cpu: "Cortex-M3" | "Cortex-M4" | "Cortex-M7" | "Cortex-M0+" | "Xtensa LX6" | "Xtensa LX7" | "RISC-V",
  maxFreq: number,            // MHz
  flashKB: number | null,
  ramKB: number | null,
  voltage: { min: number, max: number },  // V
  peripherals: { uart, spi, i2c, can, usb, eth, adc, dac },  // 数字或 boolean
  features: string[],         // ["FPU", "DSP", ...]
  priceRange: "$" | "$$" | "$$$" | null,  // 不写具体数字，只标价位段
}
```

**绝不**修改 `pinCount/package/series` 等现有字段；**绝不**改字段顺序。新字段追加在后面。
