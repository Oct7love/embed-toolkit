## Q1 数据形态演进

**体积估算**（基于现有 10 题样本测算）：

| 维度 | 10 题 × 2 语言（当前）| 50 题 × 10 语言（目标） | 倍数 |
|------|:---:|:---:|:---:|
| 单题平均 lines | ~63 行 | ~330 行（含 8 个新语言代码 + 元数据不变）| 5.2× |
| 总 lines | 633 | ~16,500 | 26× |
| 源码 KB | ~30 | ~800 | 27× |
| Gzip 后 JS bundle | ~10 KB | ~250-400 KB（取决于是否树摇）| 25-40× |

直觉：**单文件 400KB gzip 是 Next.js client bundle 的红线区**。列表页只显示 id/title/difficulty/tags，**根本不需要加载 approach/code 数据**——这是关键洞察。

**3 选项对比：**

| 选项 | 列表页首屏 | 详情页 | 维护成本 |
|------|:---:|:---:|:---:|
| (a) 单文件 `data.ts` | ~400 KB gzip | 同左 | 低 |
| (b) 按难度拆 details + 共享轻量 index | ~15 KB gzip | 按 difficulty 懒加载 50-150 KB | 中（4 个文件）|
| (c) 每题一个文件 + index | ~15 KB | 单题 ~10 KB | 高（51 个文件，import 麻烦）|

→ **倾向 (b)**：

```
lib/leetcode-hot100/
  index-data.ts                # 50 题 lightweight metadata（id/slug/title/diff/tags/desc/officialUrl/keyPoints/complexity），约 800 行 / 50 KB / 15 KB gzip
  details/
    easy.ts                    # ~10 题的 approach + 10 语言代码
    medium.ts                  # ~28 题
    hard.ts                    # ~12 题
  index.ts                     # 公共 API：filterByDifficulty / getProblemById / getDetailByDifficulty 等，detail 用 dynamic import
```

详情页 `dynamic(() => import("./details/${difficulty}"))` 按需加载。用户浏览 easy 题永远不会下载 hard 题代码。

理由：
- 列表页首屏 ≤ 50KB（远超目标 < 50KB 增量约束有挑战，但通过 lightweight index 能压住）
- 详情页 lazy load 自然分片
- 4 个文件比 51 个易管理

## Q2 Shiki 语言懒加载

当前：`createHighlighter({ themes, langs: ["cpp","python"] })` 一次性预载。

10 语言一次性预载会让 shiki 初始化时同时下载 10 个 grammar tarball（~50KB/lang × 10 = 500KB），用户首次进详情页加载时间显著变长。

**方案：lazy load per-language grammar**

```ts
let highlighterPromise: Promise<Highlighter> | null = null;
const loadedLangs = new Set<HighlightLang>();

function getHighlighter() {
  if (!highlighterPromise) {
    highlighterPromise = createHighlighter({
      themes: ["github-dark"],
      langs: [],   // 启动时不预加载
    });
  }
  return highlighterPromise;
}

export async function highlightCode(code, lang) {
  const h = await getHighlighter();
  if (!loadedLangs.has(lang)) {
    await h.loadLanguage(lang);
    loadedLangs.add(lang);
  }
  return h.codeToHtml(code, { lang, theme: "github-dark" });
}
```

效果：
- 首次进详情页：shiki core + oniguruma WASM ~200KB（一次性）+ 当前选中 lang ~50KB
- 切换到新 lang：~50KB grammar
- 再切回已加载 lang：0 KB

shiki 的 `langs` API 支持运行时 `loadLanguage(lang)`，已确认。

## Q5 质量守卫

400 代码块的现实风险：
- 我会在某个冷门语言（Rust/Kotlin/Swift）犯语法错或概念错
- 测试只能验证"shape"（如包含 `class Solution`），无法验证"语义正确"
- 没有本地编译器跑全部 400 段，最终质量靠人工抽查

**采纳的守卫**（按用户清单）：

1. **每语言 entry-point shape 测试**：每题 × 每语言断言一个特征字符串
   - C: 至少含 `int main` 或一个 `(...){` 函数体
   - C++: `class Solution`
   - Python: `class Solution`
   - Java: `class Solution` + `public`
   - JavaScript: `var \w+ = function` 或 `function \w+`
   - TypeScript: `function \w+` + 类型注解
   - Go: `func ` + Solution 相关函数名
   - Rust: `impl Solution {` 或 `pub fn`
   - Kotlin: `class Solution` + `fun`
   - Swift: `class Solution` + `func`

2. **现有不变量保留**：description ≤ 120、不抄题面 9 条 regex、approach ≥ 3 段、官方 URL 格式
   - 这些已经是 v1.5.0 的强制 lint，自动覆盖新增题

3. **分批提交**（按用户要求）：
   - **内部阶段 A** = 1 commit：升级数据结构 + 新增 8 语言 grammar 支持 + 现有 10 题补 8 语言 + UI 5tab → 10tab
   - **内部阶段 B** = 4 commits：每批 10 题 × 10 语言
   - 每批后跑 `pnpm test + pnpm build`，**任一失败立即停止汇报**

4. **抽查报告**（按用户要求）：B 阶段完成后，列出 3 道不同难度题的 Rust + Kotlin + Swift 实现，并坦白评估"是否能在标准编译器跑通"
   - **承诺不掩盖问题**：发现的语法/语义疑虑明示标 TODO，不假装完美

5. **禁止用一个超大 prompt 一次性生成 400 代码块**：每批 10 题分独立 commit，每批集中精力

**额外不做**：
- 不引入 LeetCode CLI / 在线编译器 API 验证
- 不做"代码运行次数统计"等 scope creep
- 不为"未来可能的语言"预留架构
