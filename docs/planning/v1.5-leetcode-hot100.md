## 2. 假设

- A1：10 题 MVP 数据直接放 `lib/leetcode-hot100/data.ts`（单文件 TS const 数组），约 600-900 行。扩到 100 题时再按难度拆成 `easy.ts / medium.ts / hard.ts` 懒加载。
- A2：代码（C++ / Python）内嵌为 TS 模板字符串字段，不拆到独立 .cpp / .py 文件（会产生 200+ 小文件管理麻烦）。
- A3：**版权安全**——题目类型描述自改写 50 字左右，不复制 LeetCode 官方题面。每题附 `leetcode.cn` 链接引导去官方读原题。
- A4：完成状态仅 localStorage（via zustand persist），不做导入/导出 JSON。
- A5：MVP 只支持 C++ / Python 两语言，C / TypeScript 留给后续版本。
- A6：语法高亮默认不上。用现有 CodeBlock + `font-mono` 显示纯文本代码。见 Q3 供选。

## 3. 开放问题（需你拍板）

### Q1 — 100 题数据最终形态（影响 MVP 目录结构）

| 选项 | 结构 | MVP 成本 | 扩到 100 成本 |
|------|------|:--------:|:-------------:|
| **(a) 单 TS 文件** | `lib/leetcode-hot100/data.ts` 导出 `const PROBLEMS: Problem[]` | 最低 | 100KB 单 chunk 首屏稍重 |
| **(b) 按难度拆 JSON 懒加载** | `public/leetcode/easy.json / medium.json / hard.json` + dynamic fetch（仿 chips 模式）| MVP 额外成本：写 loader | 扩到 100 零改动 |
| **(c) 每题一个 .ts 文件聚合** | `lib/leetcode-hot100/problems/001-two-sum.ts` 等 100 个文件 | 最高，目录爆炸 | 同 |

→ **倾向 (a) → 扩到 50+ 时迁移到 (b)**：MVP 阶段文件越少越好；扩到 100 时再按难度拆公共 JSON 懒加载，迁移成本约 30 分钟，值得推迟。请确认。

### Q2 — 多语言 tab 的 state 范围？

| 选项 | 行为 |
|------|------|
| **(a) 全局偏好**（zustand persist）| 用户在任一题选了 C++，下一题默认还是 C++ |
| (b) 每题独立 | 每题各自记住上次看的语言 |
| (c) 不记忆 | 每次默认 C++，切了就切了 |

→ **倾向 (a)**：用户通常固定用 1 种语言刷，跨题切换最省操作。一个 `preferredLang: "cpp" | "python"` 字段即可。请确认。

### Q3 — 语法高亮？

| 选项 | 包体积 | 行为 |
|------|:------:|------|
| **(a) 不上**，用现有 CodeBlock | 0 | 代码无颜色，只等宽字体 |
| (b) Prism.js via npm | ~50KB gzip | 2 行引入，dynamic import 只在 leetcode 页面载入 |
| (c) highlight.js via npm | ~60KB gzip | 同 |
| (d) Shiki | ~3-5MB | 最美但太重 |
| (e) 手写 C++ / Python regex 高亮器 | ~80 行代码 | 无外部 dep，只为这俩语言 |

→ **倾向 (a) 不上**：karpathy 原则"不引入未请求的依赖"；MVP 验证价值更重要；颜色可以之后加。代码区改大字号 + 明显 padding 弥补可读性即可。**如果你觉得学习工具没高亮太丑，我切 (e) 手写 regex**。请挑一个。

### Q4 — 进入详情页的 URL 形式？

| 选项 | URL |
|------|------|
| **(a) Query**：`/tools/learning/leetcode-hot100?id=1` | 方案 a，单 page，state driven |
| (b) Nested route：`/tools/learning/leetcode-hot100/1-two-sum` | 方案 b，用 Next.js dynamic route |

→ **倾向 (a) Query**：MVP 单页切换，不做真 SSR 路由（100 题一个页面太累）。扩到 100 题时再考虑是否改 (b)。请确认。

### Q5 — 完成状态粒度？

候选：
- (a) 布尔 `completed: boolean` —— 只标记"做过/没做过"
- (b) 三态 `status: "unstarted" | "reviewed" | "solved"` —— 区分"看过解析"和"自己做出来"
- (c) 打星 0-3

→ **倾向 (a) 布尔**：MVP 最小。三态能做但不是核心价值。请确认。

### Q6 — 列表页视图？

- 表格（题号 / 标题 / 难度 / 标签 / ✓）
- 卡片网格（更友好但占空间）
- 两者切换按钮

→ **倾向 表格单视图**：刷题列表就该是表格，一屏尽量多的题。请确认。

## 4. 数据 schema 定义（待 Q1-Q5 确认后固化）

```ts
// types/leetcode-hot100.ts
export type Difficulty = "easy" | "medium" | "hard";
export type Language = "cpp" | "python";

export interface Problem {
  id: number;                  // LeetCode 官方编号，如 1 / 3 / 5
  slug: string;                // URL slug，如 "two-sum"
  titleZh: string;
  titleEn: string;
  difficulty: Difficulty;
  tags: string[];              // ["数组", "哈希表"]
  /** 50 字内自改写的类型描述，不复制官方题面 */
  description: string;
  /** LeetCode 中文站链接 */
  officialUrl: string;
  /** 2-3 段核心思路 Markdown（已内联为字符串） */
  approach: string;
  solutions: Record<Language, { code: string; comment?: string }>;
  complexity: { time: string; space: string };  // "O(n)" / "O(1)"
  keyPoints: string;           // 关键考点 1-2 句
}
```

## 5. 简化方案考察

### S1：要不要做 "每题下方留笔记"？
→ 不做。localStorage 存笔记字符串是个兔子洞（同步到设备、markdown 渲染、导出...）。**MVP 绝不做**。

### S2：要不要做统计（完成率 / 按难度进度条）？
→ 做**最小版本**：列表页顶部显示 `3 / 10 ✓` 进度文字一行。不做图表、不做按难度细分进度条。扩到 100 再考虑。

### S3：要不要支持搜索？
→ MVP 10 题不做搜索。扩到 50+ 时加。

### S4：要不要排序？
→ MVP 按题号升序，不可切换。扩到 100 再加"按难度/按标签"排序。
